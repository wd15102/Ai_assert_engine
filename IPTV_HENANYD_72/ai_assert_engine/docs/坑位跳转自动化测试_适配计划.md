## 2. 方案选型：专用测试频道

### 2.1 方案对比

| 维度 | 方案 A：全量扫描 | 方案 B：专用测试频道 ✅ |
|------|----------------|----------------------|
| 范围 | 所有频道所有坑位 | 只测跳转测试频道 |
| 验证逻辑 | 按 action 类型分支（27 种） | 每模板固定一种类型（简单） |
| 维护成本 | 需适配 27 种 action | 只配几种核心跳转 |
| 执行时间 | 长（全量坑位） | 短（每模板 1-3 个坑位） |
| 覆盖率 | 理论全量 | 核心场景覆盖 |
| 扩展性 | 自动发现新坑位 | 新增跳转类型需加模板 |
| 测试桩 | 复用现有 | **需用户新建专用测试桩** |

### 2.2 选定方案 B

**核心思路**：创建一个独立的「跳转测试频道」，每个模板对应一种跳转类型，坑位标题明确标注跳转目标。

**优势**：
- 聚焦核心跳转场景，不与其他检查混在一起
- 每模板只测一种跳转类型，验证逻辑简单
- 新增跳转类型只需加新模板
- 独立运行，不影响现有模板巡检

### 4.3 验证策略（按模板名分支）

```python
# jump_verifier.py 核心逻辑
def verify_jump(template_name: str, card_title: str, screenshot: str) -> dict:
    """根据模板名自动选择验证策略"""
    if '直播' in template_name:
        return verify_live(screenshot)      # AI: 是否直播画面
    elif '视频' in template_name:
        return verify_video(screenshot)     # AI: 是否视频播放
    elif 'App' in template_name:
        return verify_app(screenshot)       # AI: 是否App启动
    elif '详情' in template_name:
        return verify_detail(screenshot)    # AI: 是否详情页
    elif 'Web' in template_name:
        return verify_web(screenshot)       # AI: 是否Web页
    else:
        return verify_generic(screenshot)   # AI: 是否正常页面
```

### 4.4 各类型验证方法

#### 直播跳转验证

| 验证项 | 方法 | 通过条件 |
|--------|------|---------|
| 画面内容 | 截图 + AI 视觉判定 | AI 返回"正在播放直播画面，无黑屏" |
| 全屏状态 | 截图检测：画面是否铺满全屏 | 无侧边列表、无坑位布局 |
| 播放稳定性 | 等待 3s 二次截图 → SSIM 对比 | SSIM < 0.95（画面在动） |

#### 视频全屏播放验证

| 验证项 | 方法 | 通过条件 |
|--------|------|---------|
| 视频加载 | 等待 5s → 截图 AI 判定 | AI 返回"视频正在播放，无黑屏/缓冲圈" |
| 播放 SSIM | 间隔 2s 双截图 → SSIM 对比 | SSIM < 0.95（画面变化） |
| 全屏状态 | 截图检测：画面占比 > 80% 屏幕 | 无模板/坑位可见 |

#### 第三方 App 跳转验证

| 验证项 | 方法 | 通过条件 |
|--------|------|---------|
| App 启动 | `adb shell dumpsys activity activities` 检查前台包名 | 前台 App 包名 = `appPackage` |
| 页面非空 | 截图 AI 判定 | AI 返回"App 已启动，非白屏" |

#### 详情页/Web 页验证

| 验证项 | 方法 | 通过条件 |
|--------|------|---------|
| 页面到达 | `dumpsys activity` 检测目标 Activity | Activity 名称匹配 |
| 页面非空 | 截图 AI 判定 | AI 返回"页面已加载，非白屏" |
| Logcat 无崩溃 | `LogcatWatcher` 事件检查 | 无 FATAL / CRASH |

## 6. 实现计划

### Phase 1：基础设施（~1.5 天）

**目标**：能解析测试桩中的跳转配置 + 能定位坑位坐标 + 能点击跳转

| 任务 | 模块 | 产出 |
|------|------|------|
| 1.1 跳转配置解析 | `jump_action_parser.py` | 从 `datas[*].defaultActions` 提取 `JumpAction` 列表 |
| 1.2 坑位坐标定位 | `card_locator.py` | 从 XML 解析 `clickable=true` 的 bounds 列表 |
| 1.3 跳转执行器 | `jump_executor.py` | `click_card` + `wait_for_jump` + `go_back` |
| 1.4 测试桩索引扩展 | `stub_parser.py` | 新增 `extract_jump_actions(stub_path)` 方法 |

**关键决策 1.1**：坑位匹配策略

测试桩 `datas[*]` 和 XML `clickable` 节点如何关联？
```
方案 A：按序号匹配 — datas[0] → 第 1 个 clickable 节点（简单但不稳定）
方案 B：按 title 匹配 — datas[*].title  vs  XML content-desc（需 OCR）
方案 C：按坐标最近邻 — 不用关联，直接在 XML 中找 clickable 点击
```

**暂定方案 A**：按序号匹配。同一模板内，测试桩 `datas` 的顺序与 UI 渲染顺序基本一致。

**关键决策 1.2**：测试桩中 `defaultActions` 为空的坑位处理

部分 `datas[*].defaultActions` 是 `null` 或空对象 → 该坑位无跳转行为 → 跳过。

## 8. 关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 坑位序号与 UI 渲染不一致 | 点击到错误坑位 | 标题 OCR 辅助匹配；可配置忽略顺序 |
| 直播/视频加载慢 | SSIM 判定超时 | 长 timeout + 缓冲圈检测（AI 判定"正在加载"≠失败） |
| 第三方 App 无法返回 | 流程卡死 | 兜底方案：HOME 键 → 重新进入频道 |
| 坑位验证耗时长 | 全量测试时间爆炸 | 可配 `JUMP_CHECK_CARD_LIMIT` 限制每模板坑位数 |
| SSIM 误判（静止画面=播放失败） | 假阴性 | 结合 AI 视觉判定（"这是直播画面"）做二次确认 |
| 广告/弹窗干扰 | 跳转后弹出广告遮挡 | 增加"关闭弹窗"步骤（检测到广告 UI → 点击关闭） |
| 模板内嵌套模板（template_data_page） | 跳转后仍是模板页而非播放器 | 按 `action` 类型分支验证，不走统一逻辑 |

### 12.5 推荐方案：被动拦截（方案 A）

```
2. 进入机顶盒首页
3. 循环：
   a. 识别当前频道名（OCR / EPGIndex 解析）
   b. 等待 GetInitMetaData 响应被拦截
   c. CharlesStubProvider 缓存该频道的 stub dict
   d. DPAD_DOWN 切换到下一个频道
4. 所有频道拦截完毕后，Charles 停止抓包
5. 复用现有引擎逐频道执行测试
```

### 12.6 整体架构：测试桩 vs 生产环境

```
测试桩驱动模式                          生产环境驱动模式
─────────────────                      ──────────────────────
                                       CharlesStubProvider（新）
                                           │
    ┌──────────────┐                       │
    │ checkpoints.json │（频道列表）          ▼

           │                               │
           ▼                               ▼
    stub_parser.parse()             stub_parser.parse()（复用）

           │                               │

           │                               │
           ▼                               ▼
    JumpCheckEngine                 JumpCheckEngine（复用）
           │                               │
           ▼                               ▼

```

### 12.7 生产环境额外考量

| 维度 | 测试桩模式 | 生产环境模式 |
|------|-----------|------------|
| 频道发现 | checkpoints.json 预配置 | EPGIndex 接口动态获取 |
| 模板结构 | 本地 JSON 文件 | GetInitMetaData 实时拦截 |
| 验证基准 | checkpoints.json 专属检查点 | 通用质量规则（排版/黑屏/乱码） |
| 执行时机 | 随时执行 | 建议避开高峰期（接口响应可能限速） |
| 数据安全 | 测试数据 | 真实用户数据，截图需脱敏 |
| 基线对比 | 无 | 建议保存每次结构快照，下次对比变化 |
