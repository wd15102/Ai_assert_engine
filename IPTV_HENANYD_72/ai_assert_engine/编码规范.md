# 编码规范（强制执行）

> 本文档汇总用户明确要求的编码规则，AI 每次编码必须遵守。优先级：本文 > SOUL.md > MEMORY.md > 通用 Best Practice。

---

## 1. 需求确认流程（先问再写）

**规则：AI 在编码前必须先提问，一次只问一个问题，直到有 95% 把握理解用户真实需求。**

- 收到需求后，先评估自己对需求的理解程度
- 如果任何环节不确定，**先提问，不猜测**
- **一次只问一个问题**，等用户回答后再继续
- 确认模板：
  > "我对需求的理解是：……，对吗？"
  - 或："请问 XX 参数应该用哪个值？A 还是 B？"
- 直到确信理解无误再开始编码

---

## 2. 需求来源规约（不得擅自加逻辑）

**规则：所有代码需求必须来自用户，AI 不得自行添加未经确认的变量、参数或逻辑。**

- 用户没说需要的东西，就是不需要
- 不要"我顺手帮你加个 XXX 功能"
- 不要引入用户未提及的配置项、参数、降级策略
- 如果发现某个需求描述有隐含依赖或空白，**先问用户确认再补**
- 编写代码之前 先看看是否已经有功能了 避免重复造轮子
---

## 3. 代码修改汇报规范

**规则：每次修改代码后，必须在回复中说清楚三项内容：**

```
文件：完整路径
位置：函数名 / 行号 / 代码段描述（精确到被替换的代码段名称）
效果：这个修改解决了什么问题，预期行为变化是什么
```

### 示例

```
文件：testsuites/test_直播频道.py
位置：navigate_to_next_template() 的循环判断逻辑
效果：去掉 Y 阈值 + static_count 判定后，焦点回卷不会误判'已到底'；
     改用目标模板名出现在 regions 中作为到达标志，兼容 RecyclerView wrap-around
```

## 4. Python 版本与语法兼容

**规则：项目运行环境为 Python 3.9，编码必须兼容。**

### 禁止使用的语法（3.10+ 才支持）

| 禁止写法 | 正确写法 |
|---------|---------|
| `dict \| None` | `Optional[dict]` |
| `list[str]` 作为类型标注 | `List[str]`（或 from `__future__` import annotations） |
| `str \| int` | `Union[str, int]` |
| `match/case` | `if/elif` |
| `TypeAlias` | 普通变量赋值 |
| `Parenthesized context managers` | 嵌套 `with` |

### 强制检查

`py_compile` 语法检查必须在**项目实际运行时的 Python 版本（3.9）**上执行：

```powershell
py -3.9 -m py_compile <文件名>
```

禁止用 3.11、3.12 等非目标版本的 Python 做语法检查。

## 5. 报错反馈规范

**规则：任何报错必须打印具体异常信息（traceback / 完整错误文本），禁止只给结论性总结。**

```
❌ 错误："报错了，可能是文件找不到"
✅ 正确：
   Traceback (most recent call last):
     File "xxx.py", line 10, in <module>
       open('not_exists.json')
   FileNotFoundError: [Errno 2] No such file or directory: 'not_exists.json'
```

## 6. 修改后三段自检

**规则：每次代码修改完成后必须依次执行三步自检，全部通过才算完成。**

| 步骤 | 检查内容 | 命令 |
|------|---------|------|
| ① 语法检查 | Python 语法无误 | `py -3.9 -m py_compile <文件>` |
| ② 编码检查 | UTF-8 无 BOM，无 GBK 乱码 | Notepad++ / VS Code 底部编码栏确认 |
| ③ 构建验证 | 前端构建通过（如适用） | `vite build` 等 |

**编码检查细则：**
- Python 模块必须使用 UTF-8 无 BOM 编码
- 禁止文件中混入 GBK/ANSI 编码的中文注释
- 如有任何疑似编码问题，立即修复

## 7. 测试桩文件不得修改

**规则：测试桩 JSON 文件是业务数据，AI 不得创建、编辑、修改或删除。**

- 测试桩目录：`D:\WorkCode\DP\DP_AutoTest\IPTV_HENANYD_72\测试桩\`
- 包含 520+ 个 JSON 文件 + MapLocal.xml
- **AI 只能读取测试桩做分析，不能写入**
- 测试桩变更由用户通过 Charles Map Local 或其他工具管理

## 8. 模板标题来源规范

**规则：电视上展示的模板标题来源是测试桩的 `datas[*].title`，不是 XML content-desc。**

- **禁止硬编码模板名列表**（如 `TEMPLATE_NAMES`）
- 必须从测试桩的 `templateDatas[*].datas[*].title` 动态提取
- 如果需要模板显示名称，走 `stub_parser.extract_template_info()` 或直接解析测试桩 JSON
- XML uiautomator dump 的 `content-desc` 是辅助参考，不应作为模板名称的唯一来源

---

## 9. 测试桩匹配规则

**规则：匹配测试桩时严格按规则操作，避免 ID 污染。**

- **只取顶层 `templateDatas[*].instacneId`**，绝不取子级 `datas` 中的 `defaultActions/selectedActions bindInstanceId`
- 子级引用会导致其他文件的 ID 污染索引，造成跨文件匹配混乱
- 索引构建用 `json.load` 全量解析（不要用 `f.read(N)` 截断，大文件的顶层 ID 可能在末尾）
- 同 `instacneId` 多文件时，按文件名与频道名匹配度打分排序

---

## 10. Python 文件编码规范

**规则：所有 `.py` 文件必须使用 UTF-8 无 BOM 编码。**

- 禁止 BOM 头（`\xEF\xBB\xBF`）
- 禁止 GBK / GB2312 / ANSI 编码
- 文件第一行可选加 `# -*- coding: utf-8 -*-`，但不是必需的
- 中文字符串建议使用 Unicode 转义或确保编辑器始终以 UTF-8 保存

---

## 11. Windows 环境规范

**规则：开发平台为 Windows，PowerShell 是默认 Shell。**

### 执行命令
- 优先使用 PowerShell 原生语法
- 禁止使用 CMD 命令语法（`type`, `copy`, `echo` 等）
- 禁止使用交互式命令，不使用需用户输入或确认的命令
- 禁止使用 `wait` 参数
- 启动 GUI 程序时使用 `Start-Process`

### 路径
- 使用反斜杠路径（`C:\path\to\file`）
- 使用短可执行文件名（`node`, `python3`）而非完整路径

### 文件操作
- 代码写入后不需要 AI 去运行/执行/验证运行效果
- AI 的任务是：**写代码 + 语法检查 + 编码检查**，不执行业务逻辑

---

## 12. 临时文件管理

**规则：不将调试临时文件写入项目工作目录。**

- 临时调试脚本写入 workspace 目录：`C:\Users\dong_\.qclaw\workspace-agent-ce165c29\`
- 或写入项目 `debug\` 子目录（如 `debug\_xxx.py`）
- 任务完成后应清理无用的临时文件
- 临时文件命名加下划线前缀（如 `_check_xxx.py`）以便识别

---

## 13. 容错设计准则

**规则：代码必须处理已知的异常场景。**

依据 SOUL.md 核心定位：

- **设备断连**：ADB 连接断开时应有重试或降级
- **应用崩溃重启**：被测 App 崩溃时自动检测并重启
- **命令超时**：shell 命令（如 dump uiautomator）必须设 timeout，防止卡死
- **降级链**：核心功能提供至少 A→B→C 三级降级策略
- 不要假设设备永远在线、UI 永远不变

---

## 14. 代码风格约定

- 使用 `Path` 对象而非字符串拼接路径
- 优先使用 `try/except` 而非 `if` 预检查
- 日志统一使用 `lib/logger.py` 的 `log_info/warn/error/ok` 函数
- 类型标注尽量使用 `Optional` / `Union` / `List` / `Dict`（3.9 兼容写法）
- 避免全局变量，优先使用实例属性
- 循环/递归必须设最大迭代/深度限制
